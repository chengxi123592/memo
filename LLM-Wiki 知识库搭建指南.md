
> 本指南基于 LLM 与人类协作构建个人知识库的实践经验总结，旨在提供一套可复用的方法论和操作流程。

---

## 1. 基于 LLM-Wiki 思想搭建知识库的优势

### 1.1 人机协作，分工明确

传统知识管理往往依赖人工维护，容易因精力有限而荒废。LLM-Wiki 模式将维护责任交给 LLM，人类专注于提供来源和审核：

| 角色 | 职责 |
|------|------|
| **人类** | 提供原始文档、提问、把关质量、指引方向 |
| **LLM** | 阅读理解、撰写页面、建立交叉引用、发现矛盾、保持一致 |

### 1.2 知识沉淀与复用

- **持久化存储**：知识不会随着对话结束而消失
- **结构化组织**：按工具/概念/资源分类，易于检索
- **可追溯来源**：每条信息标注出处，增强可信度
- **交叉互联**：相关页面互相引用，形成知识网络

### 1.3 边用边完善

- 每次提问都是检验知识库的机会
- LLM 回答后会询问是否将有价值的内容存入 Wiki
- 使用越频繁，知识库越丰富、越精准

### 1.4 保持一致性

LLM 负责全局维护，能够：
- 发现新旧信息的矛盾
- 统一页面格式和术语
- 及时更新过时的内容

## 2. 初次建立知识库

### 2.1 确定目录结构

```
your-wiki/
├── raw/                    # 原始文档（只读）
│   └── assets/            # 附件/图片
├── wiki/                   # Wiki 主目录
│   ├── index.md           # 内容索引
│   ├── log.md             # 操作日志
│   ├── contents/          # LLM生成内容
└── schema/
	├── llm-wiki-md/          	# llm-wiki思想指导文件
    └── llm-wiki-blogs.md       # 知识库规范文件
```

### 2.2 准备工作

将你认为重要的原始文档放入 `raw/` 目录：
- 官方文档（PDF、网页）
- 教程/博客
- 笔记/总结

然后进入这个知识库的根目录，启动智能体，设置工作目录

### 2.3 初始化 Wiki

**关键原则**：先骨架，后血肉

| 阶段     | 内容                                           |
| -------- | ---------------------------------------------- |
| 第一阶段 | 创建 `index.md`、`log.md`、各目录的 `index.md` |
| 第二阶段 | 填充工具总览页（一句话定位）                   |
| 第三阶段 | 逐个完善详细页面                               |

和你的智能体讨论着来（我用的CodeBuddy），一起制定规范

我的提示词如下：

```

2. **输出形式**：你主要用什么工具来浏览 wiki？Obsidian、纯文件浏览器、还是其他？
3. **Wiki 规模预期**：是几十个页面的小型知识库，还是可能有上百个页面的大规模系统？

**关于你的偏好：** 4. **页面类型**：你希望 wiki 包含哪些类型的页面？
- 📖 概念/术语解释页
- 🛠️ 工具使用指南/教程页
- 🔧 安装/配置参考页
- 📝 操作记录/经验总结页
- 🔗 工具对比/选型建议页

1. **组织方式**：按主题分类、按工具分类、还是按你自己的使用场景分类？
2. **中文还是英文**：Wiki 页面内容用中文还是英文？

- 还是希望我一次性处理所有现有文档
- 或者其他方式
这些问题会帮助我制定一个真正适合你的 schema 文档。

1、它扩展到更广的 LLM/AI 工具知识库

模型>>有几个点想确认：
1. **工具分类命名**：如 `Hermes-Agent`、`Ollama` 这种直接用工具名作为文件夹，还是用更泛化的名字如 `agent`、`inference`？
2. **资源链接**：要不要单独维护一个资源页，存放教程链接、官方文档等外部链接？
3. **对比功能**：以后是否需要工具对比页（如"Hermes Agent vs 其他 Agent 框架"）？

我>>1、直接用工具名 ；2、单独维护一个资源页 ；3、需要
```

## 3. 追加内容

### 3.1 摄入新文档

当你有新文档要加入知识库时：

原始文档 → 放入 raw/ → 告知 LLM → LLM 阅读分析→与你确认核心要点→更新/创建 Wiki 页面→更新 index.md 索引→在 log.md 记录本次摄入

**LLM 会做的操作**：

1. 阅读并分析文档
2. 讨论要点（确认重点）
3. 更新或创建相关页面
4. 建立交叉引用
5. 更新索引
6. 记录日志

```
我>>将我刚上传的三个pdf文件构建wiki

模型>>
好的，先查看新增的PDF文件,我需要读取这3个新的PDF文件，然后基于内容更新Wiki。
现在我已经读取了所有3个PDF的内容。让我基于这些内容来更新和补充Wiki。
需要更新和新增的内容：
更新Hermes-Agent/安装配置.md - 增加Docker容器部署
更新Hermes-Agent/辅助模型.md - 增加模型选择和配置
新增OpenClaw页面（作为独立工具）
更新对比页面
更新资源页面
更新index和log
让我开始创建这些页面。

所有PDF内容已获取。现在更新Wiki：
PDF Wiki 构建完成！
```

### 3.2 问答式补充

当你提问时：

提问 → LLM 搜索 Wiki → 综合回答（标注来源）→如有价值 → 询问是否存入 Wiki

这种方式让知识库在日常使用中自然生长。

```
我>>我想写一篇博客，主题为：Hermes避坑指南。请你给出相关素材和建议

模型>>让我先读取相关的 Wiki 页面，获取 Hermes Agent 的常见问题和实战经验。
根据 Wiki 中的内容，为你整理以下素材和建议：
Hermes 避坑指南 - 素材建议
一、避坑主题分类
···
二、核心避坑建议（建议作为博客正文）
···
三、博客结构建议
···
需要我帮你直接生成一篇完整的博客草稿吗？

我>>可以，生成markdown格式的，并且存入你的/wiki路径下的合适位置

模型>>考虑到这是一个关于 Hermes 的博客，可以放在 Hermes-Agent 目录下，或者创建一个 blogs 目录。让我放在 Hermes-Agent/blogs/ 目录下。
```

### 3.3 健康检查（Lint）

定期（或按需）进行：

- 检查过时信息
- 发现孤立页面（无交叉引用）
- 补充可添加的链接
- 验证索引链接有效性

## 4. 不断完善工作规范的流程

### 4.1 规范从何而来

规范不是一开始就定好的，而是在实践中产生：

实际操作中发现问题 → 与 LLM 讨论 → 形成共识 → 写入规范文件

```
我>>要将这个要求作为你以后工作规范：index.md中的索引必须有真实文件，并且要和其文件建立真实连接，可以跳转

模型>>规范已存入 schema/llm-wiki-blogs.md，以后创建/更新 Wiki 索引时会自动遵循此规范。
```

```
我>>将这一要求写入你的规范中：以后我需要你生成的文件，都要询问我是否要存入wiki中

模型>>已写入规范！以后生成文件时会自动提醒询问你。
```

### 4.2 规范应该放在哪里

规范文件建议放在项目目录下（如 `schema/`），包含：
- 目录结构说明
- 页面格式模板
- 工作流程描述
- 索引链接规范
- 角色分工

### 4.3 规范维护原则

| 原则 | 说明 |
|------|------|
| **实用优先** | 规范要能指导实际操作，不是纸上谈兵 |
| **持续迭代** | 发现问题就补充，不用追求一步到位 |
| **格式一致** | 规范本身也要遵循自己制定的格式 |
| **可验证** | 规范应包含检查/验证的方法 |

## 总结

LLM-wiki这套知识库构建思路，适合哪些场景。

- **适合**：
  - 垂直领域科研（攒你读过的好论文、专利、会议等）
  - 单个项目研发（攒项目规范、设计文档、用户手册、需求文档等）
  - 个人内容创作（攒素材、自己的作品、喜欢的作品等）
  - **总之，需要攒知识的场景（文档<100篇）**



- **不适合**：
  - 企业级海量知识库管理（还是乖乖搞RAG吧）
  - 简单的问答（直接问AI就行了）
